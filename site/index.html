Why developers care

One contract for school context, content, standards, outcomes, events, and credentials.

The platform makes the hard education integration questions explicit: who the learner is, where they are enrolled, what content they use, what standards it targets, what happened, what result was earned, and what data a tool may access.

Every SQL object and API field is backed by a shared dictionary entry, so a human developer or AI agent can understand the same platform contract without reading every 1EdTech specification.

Two developer layers

SQL for governed data access. JSON for web app integration.

Relational SQL Layer

Raw tables plus stable documented views for organizations, people, classes, standards, assessments, gradebook, events, LTI, credentials, lineage, and governance.

  • Internal IDs plus standard-specific identifier crosswalks; OneRoster-shaped endpoints preserve `sourcedId` as the standards-facing ID.
  • SQL comments generated from the same dictionary as API docs.
  • Tenant-owned records carry tenant boundaries; shared CASE/QTI/reference data is explicitly adopted by tenants.

JSON/HTTP API Layer

OpenAPI-first endpoints with examples and dictionary links on every field, designed for app developers and AI coding agents.

  • REST resources for roster, standards, content, assessment, gradebook, events, tools, credentials, and dictionary metadata.
  • The OneRoster demo API is runnable locally today; broader endpoint groups remain planned surfaces.
  • OAuth scopes mapped to dictionary fields and privacy classes.
  • Standard-compatible import/export and conformance surfaces.

Open hosted OpenAPI

Current hosted JSON API: GitHub Pages static JSON under `site/api/`, documented in the hosted OpenAPI page. Current local server API: the OneRoster core demo at `http://localhost:8787`, documented in the local Express OpenAPI page. The broader API groups below are the planned platform surface.

QTI and integration/governance now have generated repository schemas. They are documentation for the planned backend, not a live runtime on GitHub Pages. A Supabase hosted database target is provisioned for the OneRoster demo; schema, seed data, and read-only policies are packaged for loading.

Developer guide

What the platform provides, and what app builders still own.

Platform Provides

  • Roster, organization, course, class, and enrollment context normalized from OneRoster.
  • CASE standards and competency graph for alignment, search, reporting, and AI reasoning.
  • QTI package storage with searchable projections, validation status, accessibility metadata, and lineage.
  • Caliper event ingestion with immutable history and safer projections for analytics.
  • LTI 1.3 launch, LTI Advantage grade, roster, and deep-linking workflows.
  • Open Badges/CLR credential foundations for issuers, awards, verification, and status.

App Developer Owns

  • The learning experience, pedagogy, content authoring, tutoring logic, or assessment runtime inside the app.
  • User interface quality, accessibility of the app, and workflow design for educators and learners.
  • Clear scope minimization and data minimization: request only the scopes and fields the app needs.
  • Standard-specific certification for provider surfaces the app itself claims.
  • Customer-specific configuration, support, and implementation services.
